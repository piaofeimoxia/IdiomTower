# 成语塔防爆款化 Implementation Plan

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** 在保留横屏守城战场的前提下，把当前连续长波次 Demo 改造成可测试、可迭代的“点字施法 + 短房间 + 三选一轻肉鸽”抖音小游戏闭环。

**Architecture:** 保留 `GameBootstrap → SystemManager → EnemySystem/ViewSystem` 主链路，先把房间、字袋、奖励和随机提取为不依赖 Cocos 的纯 TypeScript 逻辑，再由 `SystemManager` 编排。抖音广告、分享和收藏通过平台适配接口接入，编辑器预览使用 Mock 实现。

**Tech Stack:** Cocos Creator 3.8.8、TypeScript、Node.js 内置测试运行器、`tsx`、抖音开发者工具。

---

## 一、实施范围与完成定义

本计划以产品方案中的 P0 为第一交付目标，P1、P2、P3 在 P0 数据达标后继续。

P0 完成必须同时满足：

- 保留 1280×720 横屏和左敌右门布局。
- 默认点击字块自动入槽，点击槽位可撤回。
- 单局包含 3 个普通房间和 1 个 BOSS 房。
- 房间结束后暂停并展示三选一奖励。
- 四个成语均能正常释放。
- 有正式胜利、失败、重开和一次复活接口。
- 首次进入 10 秒内可完成第一次“百步穿杨”。
- 纯逻辑测试、Cocos 编译、网页预览、抖音开发者工具验证全部通过。

P0 不包含真实广告结算、完整局外经济、好友服务端和自动短视频生成。这些分别属于 P1、P2。

## 二、非功能要求

### 性能

- 中端手机战斗帧率目标：平均 ≥ 50 FPS，P95 单帧耗时 < 25ms。
- 同屏普通敌人目标上限：30；极端验证：40。
- 点字到槽位反馈：100ms 内出现。
- 拼成语到技能起手：300ms 内出现。
- 房间结束到奖励面板可操作：800ms 内。

### 包体与加载

- 抖音小游戏整体包不超过平台限制。
- P0 优先保持主包简单；资源增长逼近限制时再启用 Asset Bundle 分包。
- 首次进入战斗前不加载 P2 社交素材和后续章节资源。

### 稳定性

- 奖励、失败、复活、切后台期间不能继续刷怪或攻击城门。
- 任意面板关闭后只能恢复一次游戏循环。
- 平台 API 失败不能卡死本局。

### 可维护性

- 新增纯逻辑模块不得导入 `cc`。
- 不再增加新的 `*Patch.ts`。
- 每个阶段结束必须删除本阶段已被正式实现替代的补丁。
- 配置、状态和显示文案不继续堆入一个超大类。

## 三、目标模块关系

```text
GameBootstrap
    │
    ▼
SystemManager ───────────────► ViewSystem
    │                           显示、输入、动画、面板
    ├─ RoomSystem
    │   房间状态与推进
    ├─ BagSystem
    │   字池、战斗袋、弃字、回池
    ├─ RewardSystem
    │   三选一、流派权重、奖励应用
    ├─ EnemySystem
    │   敌人移动、受伤、冻结、远程攻击
    └─ PlatformService
        ├─ MockPlatformService
        └─ DouyinPlatformService
```

`SystemManager` 是唯一可以同时调用多个系统的对象。`ViewSystem` 不直接修改玩法状态，只通过回调提交“点击字、撤回、选奖励、复活、重开”等用户意图。

**Step 2：实现 `RoomSystem`**

```text

```

- 每次恰好返回 3 个不同奖励。
- 至少 1 个字或成语相关奖励。
- 已满层奖励不再出现。
- 选择箭术后，后续箭术候选权重提高，但不是 100%。
- 固定随机种子得到固定结果。

**Step 3：改为立即生效三选一**

- 点击卡片立即完成选择。
- 不再需要反选和确认。
- 战斗只恢复一次。
- 奖励描述使用结果语言，不显示复杂公式。

**Verification**

- 每房间最多出现一次奖励。
- 快速连点两张卡只应用第一张。
- 奖励后 HUD 数值和技能效果同步变化。
- 连续 20 次固定种子模拟无空列表、无重复标题。

**Step 4：提交**

```powershell
git add assets/scripts tests
git commit -m "feat: add weighted three choice rewards"
```

### Task 10：实现首局引导与受控随机

**Files:**

- Create: `assets/scripts/core/AssistDirector.ts`
- Create: `tests/AssistDirector.test.ts`
- Modify: `assets/scripts/SystemManager.ts`
- Modify: `assets/scripts/systems/ViewSystem.ts`

**Step 1：实现首局 10 秒引导**

- 开局战斗袋固定包含“百、步、穿、杨”。
- 依次呼吸高亮正确字。
- 玩家操作后立即取消对应手势提示。
- 第二次施法不强制，只在停顿 3 秒时提示首字。

**Step 2：实现受控随机边界**

仅在以下条件干预：

- 城门低血且玩家拥有防御成语时，提高缺字出现概率。
- BOSS 冲锋前且玩家拥有控制成语时，提高最后缺字概率。
- 大量敌人聚集且玩家拥有范围成语时，提高范围成语缺字概率。

限制：

- 不直接把完整答案塞入槽位。
- 不改变已抽出的字。
- 每房间最多触发一次强辅助。
- 调试日志记录触发原因。

**Step 3：写测试**

- 辅助条件不满足时不干预。
- 同房间只触发一次。
- 固定种子可复现。
- 首局第一组成语始终可拼。

**Verification**

- 新玩家不读说明也能在 10 秒内释放。
- 熟练玩家快速点击时引导不会挡住输入。
- 辅助不会造成同字连续异常刷出。

**Step 4：提交**

```powershell

git commit -m "feat: add first run guidance and controlled assistance"
```

## 九、阶段门禁与回退规则

### 每个 Task 的合并门禁

- 自动测试通过。
- 纯逻辑类型检查通过。
- Creator 编译通过。
- 本 Task 人工验收通过。
- `git diff` 不包含无关格式化。
- 不覆盖用户已有未提交改动。

### 回退规则

- 每个 Task 单独提交，禁止把多个阶段压成一个大提交。
- 原型补丁只在等价行为验证后删除。
- 新 UI 未达标时可回退 View 层提交，不回退已通过测试的纯逻辑层。
- 平台能力出错时使用 Mock 或“功能暂不可用”降级，不回退核心局内玩法。

### 停止条件

出现以下情况时停止继续扩功能，先修复：

- 奖励或失败面板后游戏无法恢复。
- 同一奖励、复活或结算可重复领取。
- 三局内出现持续内存增长。
- 低端设备战斗持续低于 40 FPS。
- 首次施法中位时间高于 12 秒。

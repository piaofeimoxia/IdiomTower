# ADR-0002：提取纯逻辑核心并隔离抖音平台能力

## 状态

已接受

## 背景

当前主要玩法集中在 `SystemManager` 和 `ViewSystem`，并通过多个原型补丁覆盖奖励、弃字、穿透、弓兵表现和失败界面。最终行为依赖导入顺序，难以单元测试。

项目目前没有独立测试框架。广告、分享、收藏等抖音能力如果直接写入玩法类，会让编辑器预览和自动测试依赖平台环境。

## 决策

采用三层结构：

```text
ViewSystem（显示与输入）
        ↓
SystemManager（局内流程编排）
        ↓
纯 TypeScript 逻辑（房间、字袋、奖励、随机、局内状态）
```

抖音能力通过 `PlatformService` 接口隔离：

```text
PlatformService
├─ MockPlatformService：编辑器预览与测试
└─ DouyinPlatformService：抖音开发者工具与真机
```

纯逻辑模块不得导入 `cc`，使用 Node 测试运行器验证。Cocos 相关代码通过 Creator 编译、预览和抖音开发者工具验证。

## 结果

### 正面

- 房间推进、奖励生成和字循环可以稳定回归测试。
- 编辑器环境不需要模拟完整抖音 API。
- 平台接口失败时可以回退为普通奖励或重试提示。
- 后续扩展快手等平台时不污染玩法代码。

### 负面

- 初期需要增加接口和少量适配代码。
- 现有补丁需要逐步合并，不能一次性粗暴删除。

### 中性

- 保留 `SystemManager` 作为总协调器，不引入大型框架或全局事件总线。

## 备选方案

### 继续使用原型补丁

未采用。短期快，但新增房间、复活和平台能力后覆盖关系会更难追踪。

### 全量 ECS 或复杂状态管理框架

未采用。当前项目规模不需要，迁移风险和学习成本过高。
